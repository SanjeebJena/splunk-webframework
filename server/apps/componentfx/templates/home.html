Overview

This app provides a reference of the various components (contexts and controls) that are available in the new Splunk Application Framework. This app provides explanations and examples of the all of the components, demonstrates how they are used, and lists the options for each.

JavaScript vs Django

The Splunk Application Framework allows developers to create components using Django templates tags and JavaScript. The fundamental difference between these two methods is that Django tags are processed server-side, while JavaScript is run entirely on the client. No matter how a control is initially created (Django or JavaScript), the result is an object that you can interact with at run time using JavaScript.

Django

Django makes it possible to declare components in HTML pages using component tags. Tags are parsed by the server and replaced with the proper HTML elements before being sent to the client. On the client, the framework processes the resulting HTML, inserting the necessary DOM elements. This app shows how to instantiate all of the framework components. For example, a Splunk chart can be added to a page by adding a Django chart tag to that page's HTML template:

{% chart id="example-chart" contextid="example-search" %}

We'll show you the basics of how to use the Django template tags in this app, but we don't go too deep into Django. For that, go to the source (Django's website).

JavaScript

You can create and interact with the components of the Splunk Application Framework using JavaScript only, without any Django tags. You'll need to manually load the required resources though (this happens automatically when you use the Django tags, which call the componentloader). For JavaScript, the framework provides a load method that takes a list of components and loads the necessary files before calling the given callback function. The standard pattern for usage is:

//define dependencies
var deps = [
    "splunkjs.mvc/chart",
    "splunkjs.mvc/resulttable",
    "splunkjs.mvc/searchcontext",            
];

//load requirements and define a callback
splunkjs.mvc.load(deps,function() {
    //here the chart, resulttable and searchcontext are ready
})

Once the required elements are loaded, use the splunkjs.mvc.Components.create method to create components, providing the component type, a unique ID for the component, and a dictionary of component options as arguments. To display UI controls on the page, call render. For contexts, call start to start the search. Note that UI controls require a <div> tag, which is a jQuery selector that corresponds to a location (element ID) on the page for the control. This element ID is passed in with the el parameter (see below). The component type begins with "appfx-" (see the Component Reference here for a complete reference).

Here is an example that continues the previous code snippet by adding a resulttable and a searchcontext once components are loaded:

//define dependencies
var deps = [
    "splunkjs.mvc/chart",
    "splunkjs.mvc/resulttable",
    "splunkjs.mvc/searchcontext",            
];

//load requirements and define a callback
splunkjs.mvc.load(deps,function() {
    
    //create a resulttable. Note the appfx- prefix
    splunkjs.mvc.Components.create("appfx-resulttable", "resulttable1", {
        contextid: "search1",
        el: $("#divToHangOn")   //name the div to add this component to
    }).render();                //call render                

    //create a searchcontext. Note the appfx- prefix
    splunkjs.mvc.Components.create("appfx-searchcontext", "search1", {
        earliest_time: "-24h@h",
        latest_time: "now",
        search: "search index=_internal | stats count by sourcetype" 
    }).start();                     //call start
})

This pattern is used for all of the framework components.

Listening to click events

You can listen to and respond to click events on all controls in javascript as follows:

splunkjs.mvc.Components.getInstance("example-chart")
    .on("clicked:chart", function (e) {
        // your code here
    })

Information about the event is encapsulated in the e argument that is passed into this function.

Contexts

Search is a core feature of the Splunk App Framework. All of the framework UI controls can be bound to a search in various ways. Some controls are populated by searches, some modify the searches they are bound to, and some do both.

Each search is represented by a context, which manages the query, properties, operation, and lifetime of an individual search job. You can create contexts using Django template tags or JavaScript. Contexts can correspond to a saved search, or a search job you create on the fly in code or dynamically with UI controls.

Search context

A search context (declared with the search tag) represents a search job, which can be generated dynamically or hard coded. Here is an example of declaring a search context using the search tag. The second search corresponds to the search query—this query counts the events in the internal index:

<!-- django -->
{% search id="example-search" search="search index=_internal | stats count" %}

//javascript
splunkjs.mvc.Components.create("appfx-searchcontext", "search1", {
    search: "search index=_internal | stats count by sourcetype" 
}).start();

To bind a search context to a control, provide the context's unique ID in the control's contextid property. For example, to create a chart that is bound to the previous search context:

<!-- django -->
{% chart id="example-chart" contextid="example-search" type="line" %}

//javascript
splunkjs.mvc.Components.create("appfx-chart", "example-chart", {
    contextid: "example-search",
    el: $("#divToHangOn")
}).render();

Options

You can set the following properties using templates tags or JavaScript. These search context options do not affect the search job on the Splunk server.

    app: Sets the namespace (the app context) to run in.

    autostart: When true, search context automatically starts a new job when its properties have changed and when the page first loads (and the component-loader is called). When false, application code must call startSearch manually to start a search.

    cache: When true search always uses the results from a preexisting search job when possible. When false search never uses results from preexisting search jobs. When n, only search jobs that are newer than n seconds are used.

    search: The search query string.

    sid: The search job ID.

Search job options

Additional properties can be set (in Django or Javascript) that modify the search job on the Splunk instance. For example, to set the "latest_time" property:

<!-- django -->
{% search id="example-search" search="search index=_internal | stats count" latest_time="now" %}

//javascript
splunkjs.mvc.Components.create("appfx-searchcontext", "search1", {
    latest_time: "now",
    search: "search index=_internal | stats count by sourcetype" 
}).start();

For a complete list of search job options, see the SearchContext reference documentation.

Events

The following events are fired by this component:

    search:error: Fired when an error occurs.

    search:fail: Fired when the search job fails.

    search:progress: Fired to indicate search progress.

    search:cancelled: Fired when the search is cancelled.

    search:start: Fired when the search is started.

Saved search context

A saved search context (declared with the savedsearch tag) corresponds to a saved search (by providing its name) on a Splunk instance. Bind the saved search context to a UI control the same way you'd bind a search context. For example, to bind a table to a saved search called example-saved-search through a savedsearchcontext:

<!-- django -->
{% chart id="example-chart" contextid="saved-search" type="line" %}
{% savedsearchcontext id="saved-search" searchname="example-saved-search" %}

//javascript
splunkjs.mvc.Components.create("appfx-chart", "example-chart", {
    contextid: "example-saved-search",
    el: $("#divToHangOn")
}).render();
splunkjs.mvc.Components.create("appfx-savedsearchcontext", "example-saved-search", {
    searchname: "all100",
}).start();

Options

You can set the following properties using templates tags or JavaScript:

    app: Sets the namespace (the app context) to run in.

    autostart: When true, search context automatically starts a new job when its properties have changed and when the page first loads (and the component-loader is called). When false, application code must call startSearch manually to start a search.

    cache: When true search always uses the results from a preexisting search job when possible. When false search never uses results from preexisting search jobs. When n, only search jobs that are newer than n seconds are used.

    searchname: The name of the saved search on the Splunk server.

Events

The following events are fired by this component:

    search:error: Fired when an error occurs.

    search:fail: Fired when the search job fails.

    search:progress: Fired to indicate search progress.

    search:cancelled: Fired when the search is cancelled.

    search:start: Fired when the search is started.

Group context

A group context (declared with the group tag) groups (combines) other contexts. If one event or property change is applied to the group or to an individual context, the event or change is applied to all of them. You can also bind a control to multiple contexts by using a group context. For example, rather than selecting a time range for individual searches separately, apply a TimePicker control to multiple searches using a group context.

Note  Provide the names of the contexts to group before you provide the group context ID (id).

Here's an example:

<!-- django -->
{% search id="example-search1" search="search index=_internal | stats count" %}
{% search id="example-search2" search="search index=_internal | stats avg" %}
{% search id="example-search3" search="search index=_internal | stats max" %}
{% group "example-search1" "example-search2" "example-search3" id="search-group" %}

//javascript
splunkjs.mvc.Components.create("appfx-searchcontext", "example-search1", {
    search: "search index=_internal | stats count by sourcetype" 
}).start();
splunkjs.mvc.Components.create("appfx-searchcontext", "example-search2", {
    search: "search index=_internal | stats avg by sourcetype" 
}).start();
splunkjs.mvc.Components.create("appfx-searchcontext", "example-search3", {
    search: "search index=_internal | stats max by sourcetype" 
}).start();
splunkjs.mvc.Components.create("appfx-groupcontext", "example-group", {
    contexts:["example-search1" , "example-search2", "example-search3" ]
}).start();

Options

You can set the following properties using templates tags or JavaScript:

    id: Unique ID of the group context.

    [contextid]: A list of context IDs of the search contexts to be grouped.

Charts

The Chart control displays any series of data you want to plot in various configurable chart types.

Use the {% chart %} tag with the type property to specify which type of chart to display.

<!-- django -->
{% chart id="example-chart" contextid="example-search" type="line" %}

//javascript
splunkjs.mvc.Components.create("appfx-chart", "example-chart", {
    contextid: "example-search",
    type: "line",
    el: $("#divToHangOn")
}).render();

Chart types

You can use the following types of charts:

  • Area
  • Bar
  • Column
  • Line
  • Pie
  • Scatter

Area

An area chart (type="area") renders each series in a data set as a filled area with an optional line. The data table upon which the chart is structured must contain at least two columns: the first column contains the values to be plotted on the x-axis (such as _time values for a time chart), and each additional column contains a series of values to be plotted on the y-axis.

{% chart id="example-chart-area" contextid="chart-search" type="area" %}

Bar

A bar chart (type="bar") renders data as horizontal bars. The data set must contain at least two columns: the first column contains the values to be plotted on the y-axis, and each additional column contains a series of values to be plotted on the x-axis.

{% chart id="example-chart-bar" contextid="chart-search" type="bar" %}

Column

A column chart (type="column") renders data as vertical columns. The data table upon which the chart is structured must contain at least two columns: the first column contains the values to be plotted on the x-axis, and each additional column contains a series of values to be plotted on the y-axis.

{% chart id="example-chart-column" contextid="chart-search" type="column" %}

Line

A line chart (type="line") is a basic line graph that renders each series in a data set as a line with optional markers. The data table upon which the chart is structured must contain at least two columns: the first column contains the values to be plotted on the x-axis (such as _time values for a time chart), and each additional column contains a series of values to be plotted on the y-axis

{% chart id="example-chart-line" contextid="chart-search" type="line" %}

Pie

A pie chart (type="pie") renders data as a circle divided into slices. The data table upon which the chart is structured must contain at least two columns: the first column contains the labels for each pie chart slice, and the second column contains the values for those slices.

{% chart id="example-chart-pie" contextid="chart-search" type="pie" %}

Scatter

A scatter chart (type="scatter") is a Cartesian chart that renders data as scattered markers, and supports single and multiple series.

{% chart id="example-chart-scatter" contextid="example-search" type="scatter" %}

Options

Charts support the same options as the Splunk JSChart library. For a complete list of Splunk JSChart properties, see Chart and legend properties.

These properties must be set in JavaScript, for example:

splunkjs.mvc.Components.getInstance("example-chart").settings.set({
    "axisTitleX.text": " x-axis title ",
    "axisTitleY.text": " y-axis title ",
    "legend.placement": "none",
});

In addition to the charting library properties, you can set the following properties using templates tags or JavaScript:

    search: The search context of the chart.

    type: The type of chart.

    datasource: The name of the datasource to use.

Events

The following events are fired by charts:

    clicked:chart: Fired when the chart is clicked.

    clicked:legend: Fired when the legend is clicked.

Event table

The EventTable control is a fully-featured Splunk-style event table that supports pagination and variable formatting. Given a search context, the EventTable displays the events corresponding to that search.

Use the {% eventtable %} tag with the search property to specify the search context.

{% eventtable id="example-eventtable" contextid="example-event-search"%} 
<!-- django -->
{% block content %}
    {% eventtable id="example-eventtable" contextid="example-event-search"%}
{% endblock content %}
{% block contexts %} 
    {% search id="example-event-search" search="search index=_internal | head 3"%} 
{% endblock contexts%}

//javascript
splunkjs.mvc.Components.create("appfx-eventtable", "example-eventtable", {
    contextid: "example-search",
    el: $("#divToHangOn")
}).render();
splunkjs.mvc.Components.create("appfx-searchcontext", "example-search", {
    search: "search index=_internal | head 3" 
}).start();

Options

You can set the following properties using templates tags or JavaScript:

    event-datasource: The name of the complete event datasource.

    summary-datasource: The name of the complete summary datasource.

Forms

The form controls (Radio, Select, and TextBox) display common elements of web forms (a group of radio buttons, a drop-down list, and a text box). All of these controls have a val method that can be used to retrieve their current value. For example:

splunkjs.mvc.Components.getInstance("example-textbox").val();

Events

All forms controls for the change event.

    change: Fired when value changes.

Radio

The Radio control represents a radio button group. This control makes it easier to populate a radio button group from a search, but it can also be used like a standard HTML radio button group.

Use the {% radio %} tag and set the default option the in default property:

{% radio id="example-radio" default="One" %} 
<!-- django -->
{% radio id="example-radio" default="One"%}

//javascript
splunkjs.mvc.Components.create("appfx-radio", "example-radio", {
    default: "One",
    el: $("#divToHangOn")
}).render();

Options

You can set properties for radio button groups, but some can only be set in JavaScript because they take a list that cannot be declared using templates tags.

    choices: A static dictionary of options for the radio button group (must be set in JavaScript). 

    // set up choices            
var choices = [{label: " One", value: "One"},
{label:" Two", value: "Two"}, 
{label:" Three", value: "Three"}];

// assign them to the button group
splunkjs.mvc.Components.getInstance("example-radio").settings.set("choices", choices);

    search: The context that specifies the search to populate the radio button group. For this to be very useful, set valueField, labelField, or both to specify which fields in the search results to use for values and labels. See the example below for more on this.

    valueField: The search field to use for button values. If a labelField is not set, the button labels default to the valueField values.

    labelField: The search field to use for button labels. If a labelField is not set, the button labels default to the valueField values.

    default: The default selected button.

Example

In this example, a radio group is created and bound to a search context that shows all of the available indexes on the Splunk instance. The button group is created in the content block and the search context is created in the contexts block.

{% radio id="example-radio2" contextid="indexes-search" labelField="index" valueField="index"%} 
<!-- django -->
{% block content %}
    {% radio id="example-radio2" contextid="indexes-search" labelField="index" valueField="index"%}
{% endblock content %} 

{% block contexts %} 
    {% search id="indexes-search" search="| eventcount summarize=false index=* index=_* | dedup index | fields index" preview=False cache=True %} 
{% endblock contexts%}

//javascript
splunkjs.mvc.Components.create("appfx-radio", "example-radio", {
    default: "One",
    contextid: "indexes-search",
    labelField: "index",
    valueField: "index",
    el: $("#divToHangOn")
}).render();
splunkjs.mvc.Components.create("appfx-searchcontext", "indexes-search", {
    search: "| eventcount summarize=false index=* index=_* | dedup index | fields index",
}).start();

Text box

The TextBox control represents a basic text input box. The val method returns the current text.

Use the {% textbox %} tag and set the default text the in default property.

Options

Textbox only has one option:

    default: The default text.

Example

In this example, two text boxes are created using templates tags. In JavaScript, when a change event occurs, the text boxes swap their values.

{% textbox id="example-textbox1" default="type here" %} {% textbox id="example-textbox2" default="to be replaced" %}

In template tags:

<!-- django -->
{% textbox id="example-textbox1" default="type here" %} 
{% textbox id="example-textbox2" default="to be replaced" %}

In JavaScript:

//get textbox elements
var tb1 = splunkjs.mvc.Components.getInstance("example-textbox1");
var tb2 = splunkjs.mvc.Components.getInstance("example-textbox2");

//attach change events
tb1.on("change", function(){
    tb2.val(tb1.val());
})
tb2.on("change", function(){
    tb1.val(tb2.val());
})

Google Maps

The GoogleMap control binds a Google map to a search that interprets geographic data and displays it on a map. If your search includes lat and lng fields, markers are automatically displayed for those results on the map.

Example

In this example, the GoogleMap control is bound to a search that includes latitude and longitude values renamed as lat and lng. Search results (markers) are automatically displayed at their locations on the map.

{% googlemap id="example-map" contextid="example-map-search" %} 
<!-- django -->
{% block content %}
    {% googlemap id="example-map" contextid="example-map-search"%}
{% endblock content %}

{% block contexts %} 
    {search id="example-map-search" search="| inputlookup appfx.demo.dataset.csv | rename device_ip_latitude as lat, device_ip_longitude as lng | stats count by lat, lng "} 
{% endblock contexts%}

//javascript
splunkjs.mvc.Components.create("appfx-googlemap", "example-googlemap", {
    contextid: "example-map-search",
    el: $("#divToHangOn")
}).render();
splunkjs.mvc.Components.create("appfx-searchcontext", "example-map-search", {
    search: "| inputlookup appfx.demo.dataset.csv | rename device_ip_latitude as lat, device_ip_longitude as lng | stats count by lat, lng ",
}).start();

Options

The googlemap control wraps a google.maps.Map from the Google mapping libary. This allows developers to use the entire Google mapping API with the googlemap control. The map property of the googlemap control can be interacted with as any google.maps.Map. For more on the Google mapping API see the Google Maps API documentation.

Result table + paginator

The ResultTable control is a fully-featured Splunk-style results table that supports pagination (if you attach a Paginator control). Given a search context, the ResultTable displays the processed search results from that search.

Use the {% resulttable %} tag with the search property to specify the search context.

{% resulttable id="example-table" contextid="example-search" pageSize="5" %} 
<!-- django -->
{% resulttable id="example-table" contextid="example-search" pageSize="5"%}

//javascript
splunkjs.mvc.Components.create("appfx-resulttable", "example-resulttable", {
    contextid: "example-search",
    el: $("#divToHangOn")
}).render();

Options

You can set the following properties using templates tags or JavaScript:

    search: The search context of the table.

    datasource: The name of the datasource to use

    page: The zero-indexed page to set the table to.

    pageSize: The number of results per page.

    Fields: The fields to include in the table (must be set in JavaScript).

var table= splunkjs.mvc.Components.getInstance("example-table");
table.settings.set("fields", ["_time", "splunkd"]);

Events

The result table fires the following events:

    clicked:cell: Fired when a cell is clicked.

    clicked:row: Fired when a row is clicked.

Paginator

The Paginator control attaches a paginator to a result table to page through data.

Use the {% paginator %} tag with the search property to specify the search context.

Timeline

The Timeline control displays an event timeline of a given search and allows interactive modification of the search time range. This control modifies and is modified by its search context.

Use the {% timeline %} tag with the search property to specify the search context.

Options

You can set the following property using templates tags or JavaScript:

    datasource: The name of the datasource.

Time picker

A TimePicker control can be used by itself to modify the time range for a search context. This control can be used with a SearchBar control to modify the time range for the search bar's search.

Use the {% timepicker %} tag with the search property to specify the search context.

Options

You can set the following property using templates tags or JavaScript:

    preset: When set, the time picker starts with the selected time range. Possible values are:

    • rt-30s, Label: 30SecondWindow

    • rt-1m, Label: 1MinuteWindow

    • rt-5m, Label: 5MinuteWindow

    • rt-30m, Label: 30MinuteWindow

    • rt-1h, Label: 1HourWindow

    • rt, Label: AllTime(Real-time)

    • today, Label: Today

    • week, Label: Weektodate

    • bizweek, Label: Businessweektodate

    • month, Label: Monthtodate

    • year, Label: Yeartodate

    • yesterday, Label: Yesterday

    • prev-week, Label: PreviousWeek

    • prev-bizweek, Label: PreviousBusinessWeek

    • prev-month, Label: PreviousMonth

    • prev-year, Label: PreviousYear

    • ht-15m, Label: Last15minutes

    • ht-60m, Label: Last60minutes

    • ht-4h, Label: Last4hours

    • ht-24h, Label: Last24hours

    • ht-7d, Label: Last7days

    • ht-30d, Label: Last30days

    • alltime, Label: AllTime

    • custom, Label: Custom

Example

In this example, a TimePicker control is bound to the same search as a single value. As the time range changes with the time picker, the value changes based on the new time range.
